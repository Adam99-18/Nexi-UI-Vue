plaintext










## 三、Agent 自动测试规则（逐个测试组件）

### 1. 组件遍历规则

- 读取组件库，**获取所有组件名称**
- 遍历顺序：按字母顺序 / 按导出顺序，**一个组件测试完成再进行下一个**
- 每个组件独立测试：**创建单独测试页面，不污染其他组件**

### 2. 渲染测试（必测项）

```
测试动作：
1. 在Demo中import组件
2. 以最简方式挂载：<XXX />
3. 等待DOM渲染完成
4. 检查是否出现：
   ✅ 白屏
   ✅ 控制台报错（warn/error）
   ✅ 组件未定义
   ✅ 渲染空节点
```

### 3. 样式测试（必测项）

```
测试动作：
1. 检查DOM是否有class，但无样式
2. 检查文字/按钮/布局是否错乱
3. 检查是否未引入组件库CSS
判定规则：
- 无样式 = 样式缺失 → 必须修复
- 样式错乱 = 样式不完整 → 必须修复
```

### 4. 功能测试（核心项）

```
测试动作：
1. 点击、输入、切换、展开等基础交互
2. 传入必选props：label/value/disabled等
3. 触发事件：click/change/close等
判定规则：
- 无响应 = 功能异常
- 报错 = 代码异常
- 控制台红色错误 = 必须修复
```

### 5. 异常判定标准（Agent 只要命中就标记为「需修复」）

1. 控制台 **任何红色报错**
2. 组件 **无法渲染**
3. 只有结构 **没有样式**
4. 提示 `xxx is undefined` / `Failed to resolve component`
5. Props 类型错误、插槽不显示、事件不触发
6. 构建失败、导入失败

## 四、Agent 自动修复规则（按优先级执行）

### 优先级 1：导入 / 渲染失败修复

```
1. 检查组件库是否正确导出组件
2. 检查Demo是否正确import
3. 检查组件name是否正确
4. 检查组件是否export default
5. 修复后重新测试
```

### 优先级 2：样式缺失修复

```
1. 优先检查：Demo是否全局引入组件库样式
   - import '你的组件库名/dist/style.css'
2. 组件无独立样式文件 → 自动创建：
   - 组件目录下新建 style.css
3. 自动补基础样式：
   - 布局、边距、字体、边框、背景、hover态
4. 样式冲突 → 自动加命名空间（组件名作为class前缀）
```

### 优先级 3：Props / 属性缺失修复

```
1. 控制台提示 prop 未定义 → 自动在组件defineProps中添加
2. 提示类型错误 → 自动补全类型（String/Number/Boolean/Array）
3. 必选prop无默认值 → 自动补默认值
```

### 优先级 4：插槽 / 事件修复

```
1. 插槽不显示 → 检查组件是否写 <slot />
2. 事件不触发 → 检查是否定义 defineEmits
3. 自动补全常用事件：click / change / close / confirm
```

### 优先级 5：类型 / TS 修复（如有 TS）

```
1. 自动生成组件类型定义文件 xxx.d.ts
2. 自动补全volar插件支持配置
```

## 五、组件库源码修改规则（Agent 必须遵守）

1. **不删除原有逻辑**，只做增量修复

2. 每个组件单独提交，提交信息格式：

   ```
   fix(组件名): 修复渲染失败/补样式/补props
   ```

3. 样式必须写在**组件独立 CSS 文件**中，不写全局污染样式

4. 所有修复必须**可回滚、可验证**

5. 保持原代码风格：缩进、命名、结构不变

## 六、版本更新与发布规则（自动化发布标准）

### 1. 版本号升级规则（语义化版本）

- 仅修复 bug、补样式 / 属性 → 

  补丁版本 +1

  ```
  1.0.0 → 1.0.1
  ```

- 新增功能、增强组件 → 

  次版本 +1

  ```
  1.0.1 → 1.1.0
  ```

### 2. 发布前检查

1. 本地构建成功：`npm run build`
2. Demo 本地引入修复后的组件可正常运行
3. 无控制台报错
4. 样式完整、功能正常

### 3. 发布命令（Agent 自动执行）

```
npm run build
npm version patch  # 自动升级小版本
npm publish
```
